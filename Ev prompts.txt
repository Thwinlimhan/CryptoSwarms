## 01 · Core EV Calculator

**Tag:** `FOUNDATION`  
**Depends on:** Nothing — this is the root module  
**Used by:** All other modules

### Prompt

```
Build a production-grade Expected Value Calculator module for a probabilistic decision-making app.

CONTEXT:
The user inputs multiple outcome scenarios. The app calculates whether a decision
has positive or negative expected value.

FORMULA:
EV = Σ (probability_i × outcome_i)
Where outcomes are signed: positive for gains, negative for losses.

REQUIREMENTS:
1. Dynamic scenario builder — user can add/remove outcome rows (min 2, no max)
2. Each row has: label (text), probability (%), outcome ($)
3. Live validation: probabilities must sum to 100% (show running total with color feedback)
4. Real-time EV calculation updating as user types
5. Decision output: large bold "TAKE IT" (green) or "SKIP IT" (red) based on EV sign
6. Breakdown table showing each scenario's weighted contribution
7. Summary stats: total upside, total downside, risk/reward ratio

UI BEHAVIOR:
- Probabilities auto-normalize on blur (optional toggle)
- Outcome field accepts negative numbers for loss scenarios
- EV displayed prominently with +/- sign and color coding
- Export result as shareable JSON blob

EDGE CASES:
- Handle 100% probability (deterministic, no uncertainty)
- Warn if single scenario has >90% probability
- Prevent submission if probabilities don't sum to ~100% (±0.5% tolerance)

TECH: React with useState/useReducer, Tailwind for layout, recharts for outcome
distribution bar chart.
```

## 02 · Kelly Criterion Position Sizer

**Tag:** `RISK SIZING`  
**Depends on:** Module 01 (EV must be positive before sizing)  
**Used by:** C1 Trade Setup, C8 Master Dashboard

### Prompt

```
Build a Kelly Criterion Position Sizing tool for a trading/betting probability app.

CONTEXT:
User has a +EV opportunity. They need to know HOW MUCH of their bankroll to risk.

FORMULA:
f* = (b × p - q) / b
Where:
  b = net odds (payout per $1 risked)
  p = probability of winning
  q = 1 - p

KELLY VARIANTS to implement:
  Full Kelly  → f*
  Half Kelly  → f* / 2  (recommended for most)
  Quarter Kelly → f* / 4 (conservative, used by pros)

INPUTS:
1. Win probability (slider 0–100%)
2. Net odds b (e.g. 3 means you win $3 for every $1 risked)
3. Total bankroll ($)
4. Kelly multiplier selector: Full / Half / Quarter (radio or segmented control)

OUTPUTS:
1. f* percentage — fraction of bankroll to bet
2. Dollar amount to bet
3. Remaining bankroll after bet
4. Probability of ruin estimate: P(ruin) ≈ ((1-p)/p)^(bankroll/bet)
5. Warning banner if Full Kelly f* > 25% (psychologically brutal zone)

VISUALIZATION:
- Animated bankroll meter showing bet slice vs. reserve
- Comparison table: Full vs Half vs Quarter Kelly side-by-side
- Growth trajectory chart: simulated 50-bet compounding curve for each
  Kelly variant using recharts LineChart

EDGE CASES:
- If EV is negative (b×p < q), show "DO NOT BET — Negative EV" and disable output
- If f* > 1 (overbetting signal), cap at 100% and flag as extreme
- Handle b = 0 (graceful error)

TECH: React, Tailwind, recharts for simulated growth chart.
```

## 03 · Bayesian Belief Updater

**Tag:** `INFERENCE`  
**Depends on:** Nothing (standalone)  
**Used by:** C4 On-Chain Signals, C5 Regime Detector, C8 Master Dashboard

### Prompt

```
Build a Bayesian Belief Updater module for a probabilistic reasoning app.

CONTEXT:
The user starts with a prior belief about something being true. They receive new
evidence. The app updates their belief using Bayes' Theorem.

FORMULA:
P(A|B) = P(B|A) × P(A) / P(B)
Where:
  P(A)    = prior probability (initial belief)
  P(B|A)  = likelihood (how likely is the evidence if A is true)
  P(B)    = marginal probability of the evidence
  P(A|B)  = posterior (updated belief)

MULTI-STEP UPDATING:
After each evidence update, the posterior becomes the new prior.
Support a chain of up to 10 evidence updates.

INPUTS per evidence step:
1. Evidence label (text, e.g. "Candidate crushed technical interview")
2. Likelihood P(B|A): probability of seeing this evidence IF hypothesis is true (slider)
3. Base rate P(B): how common is this evidence overall (slider)

GLOBAL INPUT:
- Prior belief P(A): starting probability (slider 0–100%)
- Hypothesis label (text, e.g. "This trade setup has real edge")

OUTPUTS:
1. Updated posterior after each evidence step
2. Evidence log: table showing each step → prior → evidence → posterior
3. Net belief swing: delta from original prior to current posterior
4. Confidence level label: Weak (<40%) / Moderate (40–65%) / Strong (65–85%) / Very Strong (>85%)

VISUALIZATION:
- Animated probability gauge that smoothly transitions on each update
- Timeline chart (recharts) showing belief evolution across all evidence steps
- Color-coded evidence list: green if evidence increased belief, red if it decreased

PRESETS (quick-start examples):
- Trade signal validation
- Market regime assessment
- Investment thesis update
- Counterparty risk evaluation

TECH: React, recharts AreaChart for belief timeline, Tailwind.
```

## 05 · Variance & Sample Size Analyzer

**Tag:** `STATISTICS`  
**Depends on:** Module 01 (trade return data)  
**Used by:** C7 Backtest Validator, Module 08 Calibration Tracker

### Prompt

```
Build a Variance & Sample Size Analysis module for a probabilistic decision app.

CONTEXT:
Users often misjudge their skill because they draw conclusions from tiny samples.
This module quantifies uncertainty and shows when results become statistically meaningful.

COMPONENT 1 — CONFIDENCE INTERVAL CALCULATOR
  Inputs:
    - Observed win rate (%)
    - Sample size (n)
    - Confidence level (90% / 95% / 99%)
  Outputs:
    - CI: [lower bound, upper bound]
    - Width of interval
    - Interpretation: "You know almost nothing" / "Early signal" / "Solid evidence" / "High confidence"
  Formula: CI = p ± z × √(p(1-p)/n)

COMPONENT 2 — SAMPLE SIZE REQUIREMENT CALCULATOR
  Inputs:
    - Desired margin of error (e.g. ±3%)
    - Estimated true rate (%)
    - Confidence level
  Output:
    - Required sample size n
    - "You need X more observations"

COMPONENT 3 — SKILL VS LUCK DECOMPOSITION
  Inputs:
    - Win rate (%)
    - Number of trials completed
    - True edge estimate (%)
  Simulation:
    - Run 1000 Monte Carlo trials at the given edge
    - Show distribution of outcomes at n=50, n=500, n=5000
    - Highlight where the user's actual result falls
  Key insight callout: "At your current sample size, variance explains X% of your result"

COMPONENT 4 — HOW MANY TRIALS UNTIL SKILL DOMINATES?
  Input: Win rate + estimated true edge
  Output: n required for 95% confidence that performance is skill-driven
  Visual: Progress bar "You are X% of the way to statistical significance"

VISUALIZATION:
  - recharts BarChart for CI range visualization
  - recharts ScatterPlot for Monte Carlo distribution
  - Animated counter for sample size requirement

TECH: React, recharts, simple Monte Carlo in JS (no external libs needed).
```

## 06 · Prediction Market Mispricing Scanner

**Tag:** `EDGE FINDING`  
**Depends on:** Module 01 (EV formula)  
**Used by:** Standalone edge identification

### Prompt

```
Build a Prediction Market Mispricing Scanner for a probabilistic trading app.

CONTEXT:
A market prices an event at X%. The user's analysis suggests Y%. If the gap is
large enough, there's a +EV trade. This module quantifies that edge.

CORE FORMULA:
EV ratio = True probability / Market probability - 1
e.g. 0.28 / 0.12 - 1 = +133% edge

INPUTS:
1. Event description (text)
2. Market price/probability (% or decimal odds)
3. Your estimated true probability (%)
4. Reasoning log (text area — why do you think the market is wrong?)
5. Bias type selector (what's causing the mispricing?):
   - Recency bias
   - Narrative fallacy
   - Low-probability neglect
   - Overconfidence in consensus
   - Information asymmetry
   - Other (custom text)

OUTPUTS:
1. Edge % = (your_prob / market_prob - 1) × 100
2. EV per $100 bet
3. Kelly-recommended position size (auto-calculates if bankroll entered)
4. Confidence tier:
   - Edge < 10%: "Marginal — proceed with caution"
   - Edge 10–30%: "Solid edge — worth taking"
   - Edge 30–75%: "Strong edge — size up"
   - Edge > 75%: "Massive edge — double-check your math"
5. Contrarian score: how far from consensus (visual dial)

OPPORTUNITY LOG:
  - Save multiple opportunities in a list
  - Show: event | market % | your % | edge % | status (open/resolved)
  - After resolution: actual outcome, P&L, whether your estimate was calibrated

CALIBRATION TRACKER:
  After resolving 10+ predictions:
  - Plot your predicted probabilities vs actual outcomes (calibration chart)
  - Perfect calibration = diagonal line
  - Show where you're overconfident or underconfident

TECH: React, recharts ScatterChart for calibration, localStorage for opportunity log.
```

## 07 · Power Law Portfolio Simulator

**Tag:** `STRATEGY`  
**Depends on:** Module 02 (Kelly sizing)  
**Used by:** Portfolio construction decisions

### Prompt

```
Build a Power Law Portfolio Simulator for a probabilistic investing/betting app.

CONTEXT:
In environments with power law distributions (startups, crypto, options), a small
number of extreme winners dominate total returns. This module simulates that dynamic
and shows optimal portfolio construction.

POWER LAW MODEL:
P(x) = C × x^(-α)
Startup outcomes: α ≈ 1.2
Typical distribution: many small losses, rare massive winners

SIMULATOR INPUTS:
1. Number of bets/investments (1–1000, slider)
2. Cost per bet ($)
3. Win probability (%) — chance of a meaningful return
4. Payoff distribution type:
   - Fixed multiple (e.g. always 10x on wins)
   - Power law (α parameter, generates random multiples)
   - Custom: user defines probability tiers (1x, 5x, 10x, 100x with probabilities)
5. Number of simulations (100–10,000)

OUTPUTS:
1. Median portfolio outcome
2. Mean portfolio outcome (will be higher due to power law skew)
3. Probability of total loss (all bets lose)
4. Probability of 2x, 5x, 10x, 100x portfolio return
5. Expected biggest winner size
6. Optimal number of bets for maximum expected return

MONTE CARLO VISUALIZATION:
  - Run N simulations, show distribution of final portfolio values
  - recharts AreaChart: outcome distribution with percentile bands
  - Highlight: median (most likely), mean (EV), 90th percentile (lucky scenario)

DIVERSIFICATION OPTIMIZER:
  Show how returns scale with number of bets:
  - Too few: high variance, ruin risk
  - Too many: dilution
  - Optimal: the "sweet spot" highlighted on chart

PRESETS:
  - Startup angel investing
  - Crypto small caps
  - Out-of-the-money options
  - Polymarket prediction trading

TECH: React, recharts AreaChart + BarChart, Monte Carlo in vanilla JS.
```

## C1 · Asymmetric Trade Setup Calculator

**Tag:** `ENTRIES / EXITS`  
**Depends on:** Module 01 (EV), Module 02 (Kelly)  
**Used by:** C6 Live Trade Dashboard, C8 Master Dashboard

### Prompt

```
Build a professional Asymmetric Trade Setup Calculator for a crypto trading app.

CONTEXT:
Before entering, answer: Is this trade +EV after fees? How much to risk? Where
are my targets? This module answers all three.

CORE FORMULA:
EV = (win_prob × reward) - (loss_prob × risk)
Net EV = EV - (position_size × fee_rate × 2)  ← entry + exit fees

INPUTS — TRADE PARAMETERS:
1. Asset (text, e.g. BTC/USDT)
2. Direction: LONG / SHORT (toggle)
3. Entry price ($)
4. Stop Loss price ($)
5. Take Profit tiers:
   - TP1 price + % of position to close (e.g. 50%)
   - TP2 price + % of position to close (e.g. 30%)
   - TP3 price + % remaining (auto-fills to 100%)
6. Win probability estimate (%) — user's conviction
7. Portfolio size ($)
8. Risk % per trade (slider, 0.5%–5%)
9. Exchange fee (maker/taker, default 0.04%/0.06%)

CALCULATED OUTPUTS:
1. Risk in $ = |entry - stop| / entry × position_size
2. Reward per TP tier in $
3. Weighted average R:R
4. EV per trade ($) — fee adjusted
5. Position size in contracts/units
6. Liquidation price (for leveraged trades)
7. BE% — how far price moves before stop hit

TRADE QUALITY SCORE (composite):
  - R:R ≥ 2: +1 point
  - R:R ≥ 3: +1 point
  - EV > 0 after fees: +1 point
  - Win prob > 45%: +1 point
  - Multiple TP tiers: +1 point
  Score 0–5 → F / D / C / B / A / A+

VISUALIZATION:
  - Price ladder: vertical chart with entry, SL, TP1/2/3 as reference lines
  - Payoff diagram: bar chart showing $ outcome at each scenario
  - Risk/Reward dial: animated arc gauge

MULTI-SCENARIO COMPARISON:
  Build 2–3 setups side by side
  Table: EV | R:R | Quality Score | Position Size

TECH: React, recharts ComposedChart + BarChart, Tailwind, localStorage for trade log.
```

## C3 · Liquidation Map & Cascade Probability

**Tag:** `LIQUIDATION MAPS`  
**Depends on:** C1 (price levels), C5 (regime context)  
**Used by:** C8 Master Dashboard

### Prompt

```
Build a Liquidation Map & Cascade Probability module for a crypto trading app.

CONTEXT:
Liquidation clusters act as price magnets. Understanding them is a core structural edge.

MODULE 1 — LIQUIDATION ZONE BUILDER
  Inputs per zone (from Coinglass data, pasted manually):
    - Price level ($)
    - Estimated liquidation volume ($M)
    - Direction: LONG liquidations / SHORT liquidations
  
  Outputs:
    - Liquidation density map: bar chart by size and direction
    - Total long liquidations above/below current price
    - Long/Short liquidation ratio (imbalance signal)
    - Nearest major cluster: distance in % and $

MODULE 2 — CASCADE PROBABILITY CALCULATOR
  Inputs: starting price, target zone, zone volume, avg daily volume, impact coefficient
  
  Formula:
    price_impact = (liq_volume × impact_coeff) / avg_daily_volume × 100
    cascade_probability = min(price_impact / distance_to_next_zone × 100, 95%)
  
  Outputs:
    - Probability that hitting Zone A cascades to Zone B (%)
    - Expected price movement from cascade
    - Cascade chain: Zone A → Zone B → Zone C with probability decay
    - Liquidity vacuum identifier (gaps between zones)

MODULE 3 — SWEEP TARGET PROBABILITY RANKER
  Score per zone = (volume × 0.4) + (proximity_score × 0.35) + (imbalance_factor × 0.25)
  
  Output: Top 5 sweep targets ranked by probability
  If sweep probability > 60%: suggest fade setup (enter after sweep completes)

MODULE 4 — EV OF SWEEP TRADE
  Auto-generates trade params for a liquidity sweep play:
    - Entry: just beyond the liquidation zone
    - SL: beyond next cluster
    - TP: 50% retrace of sweep move (mean reversion target)
    - Win probability: derived from cascade probability score

## C7 · Strategy Backtest EV Validator

**Tag:** `BACKTESTING`  
**Depends on:** Module 05 (statistical testing), Module 08 (tracking)  
**Used by:** C8 Master Dashboard (historical edge score)

### Prompt

```
Build a Strategy Backtest EV Validator for a crypto trading app.

CONTEXT:
This module answers: "Is this backtest result a real edge or luck + data mining bias?"
Every result that passes gets elevated to live trading. Everything else is discarded.

MODULE 1 — TRADE HISTORY IMPORTER
  Input: CSV upload OR manual entry
  Required CSV columns: date | asset | direction | entry | exit | size | pnl | win (1/0) | regime_tag
  Validation: detect duplicates, missing data, date range, assets covered

MODULE 2 — CORE PERFORMANCE STATISTICS
  Basic stats:
    total_trades, win_rate, avg_win, avg_loss, R:R
    total_pnl, profit_factor, max_drawdown, max_consecutive_losses
    Sharpe ratio (annualized), Sortino ratio, Calmar ratio
  
  EV analysis:
    EV per trade = (win_rate × avg_win) - (loss_rate × avg_loss)
    Edge per dollar risked = EV / avg_loss
    Kelly fraction = win_rate - (1-win_rate)/edge_ratio

MODULE 3 — STATISTICAL SIGNIFICANCE TESTER
  Binomial test: is win_rate significantly > 50%?
    Z = (wins - n×0.5) / √(n×0.25), p-value
  
  Monte Carlo permutation test:
    Shuffle trade outcomes 10,000×
    If actual P&L > 95th percentile of random → p < 0.05 (significant)
  
  Flags:
    n_trades < 100: "INSUFFICIENT SAMPLE"
    n_trades < 30: "INVALID — DO NOT USE"

MODULE 4 — REGIME BREAKDOWN ANALYSIS
  If trades have regime tags, show performance breakdown:
    Win rate / EV / Profit factor per regime
  
  Key output: "This strategy is +EV in TRENDING BULL but -EV in CHOPPY —
               add regime filter!"

MODULE 5 — EQUITY CURVE ANALYZER
  recharts AreaChart: equity curve over time
  Drawdown periods: red shading
  Rolling 20-trade win rate: secondary axis
  
  Random equity curves overlay:
    Show 20 random-walk curves alongside actual
    If actual within random cloud → no proven edge

MODULE 6 — FORWARD WALK VALIDATION
  Split: first 70% in-sample / last 30% out-of-sample
  Compare: win_rate, EV, profit factor IS vs OOS
  
  Degradation > 50%: HIGH overfitting risk → DISCARD
  Degradation 20–50%: MODERATE — normal curve-fitting
  Degradation < 20%: LOW — robust strategy
  
  Verdict: "PROCEED WITH LIVE TRADING" / "NEEDS MORE VALIDATION" / "LIKELY OVERFIT"

VISUALIZATION:
  Equity curve recharts AreaChart
  Performance heatmap: day × hour grid (recharts or custom)
  Regime breakdown recharts BarChart
  Statistical significance bell curve with actual result marked

TECH: React, recharts, Papaparse for CSV, Tailwind, Monte Carlo in vanilla JS.
